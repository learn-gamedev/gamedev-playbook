# Этапы разработки проекта: Imperial Courier — Shih Tzu Run

Пошаговый (итеративный) план разработки игры с нуля.

**Принцип:** после каждого этапа проект **запускается и показывает что-то новое**.
Ни одного этапа "код не работает, доделаю потом". Каждый этап — это:

1. Тест (`red`) — что должно работать.
2. Минимальный код, чтобы тест прошёл (`green`).
3. Рефакторинг при необходимости.
4. Ручной прогон игры — визуальное подтверждение результата.

Каждый этап коммитится отдельно.

---

## Этап 0. Пустой проект

Разбит на 4 маленьких шага — каждый со своим результатом, чтобы даже на этом
уровне не смешивать "скелет запускается" с "появилась инфраструктура".

### Шаг 0.1. Голый скелет

**Цель:** голый скелет, который запускается.

- Репозиторий, `.gitignore`, `uv init` (создаёт `pyproject.toml`), зависимости через `uv add`: `pygame`, `pytest`, `ruff`.
- Папки: `src/imperial_courier/`, `tests/`, `assets/`, `docs/`.
  - Именно `src/imperial_courier/`, а не голый `src/` — это конвенция **src-layout**. Пакетом, который ставится через `uv sync` и импортируется (`from imperial_courier...`), должен быть `imperial_courier`, а не `src`: `src` — плохое имя пакета (неинформативное, конфликтует с другими проектами с таким же layout в одном окружении). `src/` сам по себе — просто контейнер, не пакет. Ссылка на пакет прописывается в `pyproject.toml`: `packages = ["src/imperial_courier"]`.
  - Побочный эффект src-layout: без него Python мог бы импортировать код прямо из рабочей директории (через `sys.path`), даже если пакет не установлен — это маскирует баги с путями/зависимостями. С `src/` такое исключено.
- `main.py` — тонкая точка входа, вызывает `imperial_courier.app.main()`.
- `app.py`: `pygame.init()`, окно `WIDTH×HEIGHT`, заливка цветом, `while running` с обработкой `QUIT`, `clock.tick(FPS)`.
- `settings.py`: `WIDTH`, `HEIGHT`, `FPS`, `TITLE`, базовые цвета.

**Результат:** `uv run python main.py` открывает пустое окно нужного размера и закрывается по крестику.

**Тест:** пока не требуется автотестов — нечего проверять кроме "окно открывается". Достаточно ручной проверки.

### Шаг 0.2. Иконка окна

**Цель:** у игры собственная иконка вместо дефолтной иконки Python в переключателе окон/панели задач.

- `assets/images/icon.png` — квадратное изображение (например, 64×64) с альфа-каналом.
- `app.py`: `pygame.display.set_icon(pygame.image.load(...))` — вызывается **до** `pygame.display.set_mode(...)`.
- Fallback: если файла нет — не падать, а просто не выставлять иконку (правило проекта про обработку отсутствия ассетов).

**Результат:** окно и переключатель окон (Cmd+Tab / Alt+Tab) показывают иконку игры. На macOS Dock при запуске через `python main.py` может по-прежнему показывать иконку Python — это ограничение платформы (Dock привязан к иконке процесса-интерпретатора, а не к `set_icon()`), не баг кода.

**Тест:** ручная проверка — `set_icon()` не бросает исключение ни при наличии файла, ни при его отсутствии.

### Шаг 0.3. Логгер

**Цель:** завести журналирование как сквозную (cross-cutting) инфраструктуру
до того, как появится много кода, который пришлось бы ретроактивно ею обвешивать.

- `settings.py`: `LOG_DIR`, `LOG_FILE`, `LOG_LEVEL`, `LOG_MAX_BYTES`, `LOG_BACKUP_COUNT` — константы вместо магических чисел.
- `utils/logger.py`: `get_logger() -> logging.Logger` — лениво настраивает `logging.handlers.RotatingFileHandler` (директория `logs/` создаётся автоматически, ротация по размеру с хранением `LOG_BACKUP_COUNT` бэкапов). Идемпотентность проверять по наличию уже прикреплённого `RotatingFileHandler`, а не по факту "есть хоть один хендлер" — иначе сторонний код (например, тестовый раннер) может незаметно повесить свой хендлер на тот же логгер и обмануть проверку.
- `app.py`: `get_logger().info("Game starting")` в начале `main()`, `get_logger().info("Game closing")` перед `pygame.quit()`.
- Конкретные события (переходы стейтов, Game Over, ошибки `save.json`) логируются по одной строке в тех этапах, где эти механики появляются (см. Этапы 1, 5, 12) — здесь заводится только сама инфраструктура.

**Результат:** `uv run python main.py` по-прежнему открывает окно, но в `logs/game.log` появляются записи о старте и закрытии.

**Тест:** на этом шаге ещё не пишем — сначала руками убеждаемся, что логгер работает и файл создаётся; автотесты — следующим шагом (0.4), отдельно от написания самого кода логгера.

### Шаг 0.4. Тесты на логгер

**Цель:** формализовать поведение логгера автотестами, прежде чем опираться
на него в последующих этапах.

- `tests/test_logger.py`:
  - `get_logger()` создаёт директорию `logs/`;
  - хендлер — именно `RotatingFileHandler`, с `maxBytes`/`backupCount` из `settings.py`;
  - повторные вызовы `get_logger()` не дублируют хендлер;
  - сообщение, записанное через `logger.info(...)`, реально попадает в файл.
- Изоляция теста: `settings.LOG_DIR` подменяется через `monkeypatch` на `tmp_path`, чтобы тестовые логи не утекали в реальный `logs/` репозитория.

**Результат:** `uv run pytest` — зелено; дальнейшие этапы (1, 5, 12), которые логируют свои события, могут полагаться на уже протестированный `get_logger()`.

> В коде проекта на данный момент реализован только Шаг 0.1 (голый скелет); иконка (0.2), логгер (0.3) и его тесты (0.4) — следующие по плану, ещё не реализованы.

## Этап 1. State Machine + пустое меню

**Цель:** каркас стейтов, на экране — текст меню, без логики игры.

- `settings.py`: класс `State` со строковыми константами (`MENU`, `PLAYING`, ...) — пока только `MENU` и заглушка `PLAYING`.
- `states/menu.py`: `MenuState.update(dt, events) -> str | None`, `draw() -> None`. Рисует заголовок игры и подсказку "Enter — играть".
- `app.py`: словарь `states`, диспетчер `current`, обработка `next_state`.
- По `Enter` возвращается `State.PLAYING` — но `PLAYING`-стейт пока просто рисует надпись-заглушку "Game placeholder" и клавишей `Esc` возвращает в `MENU`.
- `app.py`: `get_logger().info("State transition: %s -> %s", current, next_state)` при каждом переходе (логгер уже настроен в Этапе 0).

**Результат:** запуск → меню → Enter → заглушка игрового экрана → Esc → снова меню; в `logs/game.log` видна последовательность переходов.

**Тест:** `tests/test_app_states.py` — юнит-тест, что `MenuState.update()` при событии `KEYDOWN Enter` возвращает `State.PLAYING`, а без событий — `None`.

## Этап 2. Игровое поле со статичными объектами

**Цель:** реальный `GameState`, но без движения — статичная сцена.

- `entities/player.py`: класс `Player(pygame.sprite.Sprite)` — прямоугольник-заглушка (`image.fill`), фиксированная позиция `x=150`, `rect.bottom = GROUND_Y`.
- `states/game.py`: `GameState` создаёт `Player`, рисует линию земли (`GROUND_Y`) и один статичный `Obstacle` (тоже прямоугольник) на экране. Ничего не двигается.
- `settings.py`: `GROUND_Y`.

**Результат:** на игровом экране видно игрока, землю и одно препятствие — все статично.

**Тест:** `tests/test_player.py` — `Player()` создаётся с корректным стартовым `rect` (позиция, размер).

## Этап 3. Управление и физика игрока (прыжок, гравитация)

**Цель:** игрок реагирует на ввод, есть гравитация.

- `settings.py`: `GRAVITY`, `JUMP_FORCE`.
- `Player`: атрибуты `vy`, `on_ground`; метод `jump() -> bool` (прыжок только с земли); в `update(dt)` — гравитация, приземление о `GROUND_Y`.
- `GameState.update`: обработка клавиши прыжка → `player.jump()`.
- **Delta time:** всё движение уже сейчас умножается на `dt` (правило проекта).

**Результат:** пробел/стрелка вверх — игрок подпрыгивает и падает обратно на землю по параболе.

**Тест (red → green):**
- `test_jump_from_ground_returns_true`
- `test_jump_sets_negative_vy`
- `test_gravity_pulls_player_down` (после `update(dt)` без прыжка `vy` растёт, `rect.y` растёт, если не на земле)
- `test_player_lands_on_ground` (после падения `rect.bottom == GROUND_Y`, `on_ground == True`)

## Этап 4. Движение мира и первое препятствие с коллизией

**Цель:** ощущение раннера — мир скроллится, столкновение с препятствием что-то делает.

- `settings.py`: `PLAYER_BASE_SPEED` (пока константа, без роста).
- `entities/obstacles.py`: базовый `Obstacle(pygame.sprite.Sprite)` — `update(dt)` двигает `rect.x -= world_speed * dt`; `Vase(Obstacle)` — простейший подкласс без переопределений.
- `GameState`: группы `all_sprites`, `obstacles`; один `Vase`, который движется навстречу игроку.
- Коллизия: `pygame.sprite.spritecollide(player, obstacles, False, collide_rect_ratio(0.7))` → пока просто печатает в консоль / меняет цвет игрока на кадр.

**Результат:** ваза едет справа налево, при касании игрока происходит видимая реакция.

**Тест:** `test_obstacle_moves_left_with_world_speed`, `test_collision_detected_between_player_and_obstacle`.

## Этап 5. Механика "шерсть" (здоровье) и Game Over

**Цель:** проигрыш и рестарт — цикл раннера замкнут.

- `settings.py`: `FUR_MAX`, `FUR_START`, `FUR_DECAY_RATE`, `FUR_HIT_PENALTY`.
- `Player`: атрибут `fur`, пассивный распад `-FUR_DECAY_RATE * dt` в `update`; метод `take_damage()`.
- `Obstacle.on_hit(player)` → `player.take_damage()`.
- `states/game_over.py`: `GameOverState` — надпись "Game Over", счёт, "Enter — заново".
- `GameState.update`: при `fur <= 0` возвращает `State.GAME_OVER`; переход `GAME_OVER → PLAYING` вызывает `game_state.reset()`.
- `GameState.update`: перед `return State.GAME_OVER` — `self._logger.info("Game over: score=%d", self._score)`.

**Результат:** шерсть тает сама по себе и от удара; при 0 — экран Game Over; можно начать заново; в `logs/game.log` фиксируется счёт каждого проигрыша.

**Тест:** `test_fur_decays_over_time`, `test_damage_reduces_fur`, `test_fur_cannot_go_below_zero`, `test_game_over_triggered_at_zero_fur`.

## Этап 6. Спавнер — бесконечная генерация препятствий

**Цель:** препятствия появляются сами, бесконечно, а не руками на сцене.

- `systems/spawner.py`: `Spawner` с таймером (`SPAWN_INTERVAL_START`, `SPAWN_INTERVAL_MIN`), метод `update(dt, ...)` создаёт `Vase` по истечении интервала и сбрасывает таймер.
- Интервал спавна постепенно уменьшается от `SPAWN_INTERVAL_START` к `SPAWN_INTERVAL_MIN` с течением времени (`elapsed`).
- Рост скорости мира: `SPEED_INCREMENT` каждые `SPEED_STEP_TIME` секунд до `PLAYER_MAX_SPEED`.

**Результат:** игра становится бесконечной — вазы появляются с нарастающей частотой, мир ускоряется.

**Тест:** `test_interval_at_start_is_max`, `test_interval_decreases_with_time`, `test_interval_never_below_min`, `test_update_spawns_obstacle_when_timer_expires`.

## Этап 7. HUD — счёт и полоска шерсти

**Цель:** игрок видит прогресс и состояние здоровья.

- `systems/hud.py`: `HUD.draw(screen, player, score, high_score)` — полоска `fur` (цвет меняется по порогам), текст очков.
- `GameState`: счёт растёт от пройденной дистанции (`distance / 10`).
- `settings.py`: `HUD_BAR_*`.

**Результат:** на экране видна полоска шерсти и растущий счёт в реальном времени.

**Тест:** ресурсы HUD кэшируются при инициализации, а не создаются каждый кадр (правило проекта про производительность) — можно проверить юнит-тестом, что шрифт создаётся один раз (`load_font` вызывается один раз за сессию, не в `draw()`).

## Этап 8. Подборы (pickups) — второй тип интерактивных объектов

**Цель:** не только урон, но и бонусы.

- `entities/pickups.py`: базовый `Pickup`, подклассы `Cookie` (+очки), `Comb` (+шерсть), `Clip` (+шерсть меньше), `Scroll` (+очки, неуязвимость).
- `Player.apply_pickup(kind)` или `pickup.apply(player)` — логика бонуса.
- Спавнер расширяется вторым таймером для подборов.
- `settings.py`: `SCORE_COOKIE`, `SCORE_SCROLL`, `FUR_COMB_BONUS`, `FUR_CLIP_BONUS`, `INVINCIBLE_SCROLL_TIME`.

**Результат:** печеньки/расчёски/свитки падают вместе с вазами, дают предсказуемый эффект при подборе.

**Тест:** `test_cookie_awards_score`, `test_comb_restores_fur`, `test_fur_capped_at_max`, `test_scroll_grants_invincibility`.

## Этап 9. Разнообразие препятствий: Beam, Cat (ИИ), Gap (пропасть)

**Цель:** усложнение геймплея — не только "прыгнуть через вазу".

- `Beam(Obstacle)` — спавнится на высоте `GROUND_Y - 64 - 24`, требует приседания/прыжка через.
- `Cat(Obstacle)` — простой ИИ: движется быстрее мира (`CAT_EXTRA_SPEED`), периодически прыгает (`CAT_JUMP_INTERVAL`) — своя гравитация и `on_ground`.
- `Gap` — отдельный спрайт, отдельная группа `gaps` (не в `obstacles`!). Падение проверяется вручную: `player.rect.bottom > GROUND_Y + GAP_FALL_THRESHOLD and not on_ground` → Game Over.
- Двойной прыжок: `DOUBLE_JUMP_THRESHOLD` — доступен только при высокой шерсти; множитель прыжка `_get_jump_multiplier()` (0.5/0.75/1.0/1.4) по шкале шерсти.

**Результат:** полный набор препятствий из GDD, вертикальное разнообразие (под балкой, над пропастью, вокруг кота).

**Тест:** `test_double_jump_available_when_fur_high`, `test_no_third_jump_in_air`, `test_gap_beam_available_after_30s_in_palace` (см. существующий набор тестов проекта — они уже написаны для этой механики и могут использоваться как референс).

## Этап 10. Слой уровней (`levels/`)

**Цель:** несколько сценариев игры вместо одного жёстко зашитого.

- `levels/base.py`: абстрактный `BaseLevel` — `parallax_layers()`, `spawn_pool(elapsed)`, `speed_at(elapsed)`.
- `levels/palace.py`, `garden.py`, `mountains.py` — конкретные уровни с разными пулами препятствий и стартовой скоростью.
- Спавнер и `GameState` перестают хранить логику скорости/пула самостоятельно — берут её из `level`.
- `states/level_select.py`: экран выбора уровня, блокировка по `unlock_score`.

**Результат:** в меню появляется выбор из 3 уровней с разной сложностью и визуалом.

**Тест:** `test_only_vase_at_start_in_palace`, `test_cat_available_after_15s_in_palace` — уже присутствуют в проекте, переиспользуются как спецификация поведения.

## Этап 11. Параллакс-фон и полировка визуала

**Цель:** ощущение глубины и движения фона, разное для каждого уровня.

- `systems/parallax.py`: `Parallax` — несколько слоёв с разным коэффициентом скорости (`factor`), конфигурация приходит из `level.parallax_layers()`.
- Fallback: если PNG-ассета нет — рисуется цветная плашка-заглушка (`_make_placeholder`), игра не падает без картинок (правило проекта про fallback загрузки).

**Результат:** фон каждого уровня визуально отличается, слои двигаются с разной скоростью.

**Тест:** ручная проверка + юнит-тест на fallback (`Parallax` с несуществующим путём к картинке не бросает исключение и возвращает валидную `Surface`-заглушку).

## Этап 12. Сохранения и рекорды

**Цель:** персистентность между запусками.

- `utils/save.py`: `load_save()` / `write_save()`, `save.json` с дефолтами при отсутствии файла.
- `high_score` обновляется при Game Over, используется для разблокировки уровней (`unlock_score`).
- `utils/save.py`: при `json.JSONDecodeError`/`OSError` — `get_logger().warning("Failed to load save.json: %s", e)` перед возвратом дефолтов.

**Результат:** рекорд сохраняется между перезапусками игры, разблокирует уровни; повреждённый `save.json` не роняет игру и оставляет след в `logs/game.log`.

**Тест:** `test_load_save_returns_defaults_when_file_missing`, `test_write_save_then_load_save_roundtrip`.

## Этап 13. Пауза, звук, анимации — финальная полировка

**Цель:** довести MVP до полноценной игры.

- `states/pause.py`: `PauseState`, переход `PLAYING ↔ PAUSED` по `Esc`.
- Звуки: `_try_load_sound()` с обработкой `pygame.error` (fallback — без звука, не падать).
- Спрайт-анимации игрока (`anim_state`, `anim_frame`, кадры по фазам fur-tier).
- `utils/ui.py`: `Button`, `load_font` — кэшированные ресурсы UI.

**Результат:** игра ощущается завершённой — есть пауза, звук, анимация, а не только заглушки-прямоугольники.

**Тест:** ручная проверка UX + юнит-тесты на переходы `PauseState`.

## Как это соотносится с учебным модулем

Этапы 0–5 покрывают **итерацию 1 (MVP)** из `Модуль 3. Командный игровой проект.md`
(игровой цикл, State Machine, журналирование, игрок, одна механика, счёт).
Этапы 6–13 — **итерацию 2** (уровни/препятствия, ИИ, сохранения, звук, меню, анимации).
Этап 14 — **отладка и финализация**.

Общий принцип на каждом шаге: **red → green → рефакторинг**, и после рефакторинга —
проверка, что игра по-прежнему запускается и предыдущие механики не сломаны.
